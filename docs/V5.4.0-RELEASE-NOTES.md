# SkillFoundry v5.4.0 Release Notes

**Release Date:** 2026-04-19
**Codename:** Token Optimization

---

## Overview

v5.4.0 addresses the #1 hidden cost in AI-assisted development: every MCP tool response becomes part of the conversation history and gets re-tokenized on every subsequent turn. Research shows 98.5% of tokens go to rereading old context — this release cuts the framework's contribution to that cost with a response optimizer, concise mode for skill prompts, and persistent token tracking.

---

## New Features

### Response Optimizer — Smart JSON Compaction

All JSON tool responses now go through automatic compaction before returning to the LLM:

- **Null/empty stripping** — Removes null, undefined, and empty string fields from response payloads
- **Size-aware formatting** — Pretty-print for small payloads (<2KB), single-indent for medium (<8KB), minimal for large
- **Array truncation** — Caps arrays at 50 items with a count summary (e.g., "... and 73 more items")
- **Output truncation** — Hard cap at ~30K characters (~8,500 tokens) with a summary header noting how much was omitted

### Concise Mode — Skill Prompt Compression

Every LLM skill tool now accepts a `concise: true` parameter:

- Strips markdown code examples (replaced with `[example omitted]`)
- Collapses table rows beyond 3 (header + separator + 3 data rows)
- Removes blockquote sections (usually philosophy/context, not instructions)
- Collapses consecutive blank lines
- **Typical reduction: 40-65% fewer tokens per skill invocation**

Use this on repeated or familiar skills where the full instruction set isn't needed. The core rules and directives are always preserved.

### Token Tracker — Persistent Usage Tracking

SQLite-persisted token usage tracking that survives server restarts:

- **Per-session tracking** — Each server session gets a unique ID; all tool invocations are recorded with output token counts
- **Daily aggregation** — Query usage by date for daily cost reports
- **Budget warnings** — Automatic warnings at thresholds:
  - 100K tokens: "Use concise:true on familiar skills"
  - 200K tokens: "Consider starting fresh"
  - 500K tokens: "Start a new conversation to avoid exponential cost"
- **Cost estimation** — Calculated at Sonnet input pricing since tool outputs become input tokens on re-read

### `sf_token_report` — New MCP Tool

Real-time token usage visibility:

```json
{
  "sessionId": "sf-1713538963-a7x2k1",
  "sessionDurationMinutes": 45,
  "totalOutputTokens": 87432,
  "totalInvocations": 23,
  "estimatedCostUsd": 0.2623,
  "byTool": [
    { "toolName": "sf_architect", "invocations": 3, "totalOutputTokens": 34200, "avgOutputTokens": 11400 },
    { "toolName": "sf_coder", "invocations": 5, "totalOutputTokens": 28100, "avgOutputTokens": 5620 }
  ],
  "budgetWarning": null
}
```

Pass `date: "2026-04-19"` for daily aggregate reports.

## Database Changes

New SQLite table (auto-created at startup):

```sql
CREATE TABLE IF NOT EXISTS token_usage (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id TEXT NOT NULL,
  tool_name TEXT NOT NULL,
  output_tokens INTEGER NOT NULL,
  output_chars INTEGER NOT NULL,
  duration_ms INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

Indexes: `session_id`, `created_at`, `tool_name`.

**Migration:** None required. Table is auto-created if it doesn't exist.

## Token Savings — What to Expect

| Scenario | Before | After | Savings |
|----------|--------|-------|---------|
| Large skill prompt (e.g., /forge) | ~12,000 tokens | ~5,000 tokens (concise) | 58% |
| JSON response with nulls | ~800 tokens | ~500 tokens | 37% |
| Large scan result (200+ items) | ~15,000 tokens | ~8,500 tokens (truncated) | 43% |
| 30-tool session (cumulative) | ~180K output tokens | ~95K output tokens | 47% |

The key insight: these savings compound. Every token saved in a tool response is one fewer token re-read on every subsequent turn. A 500-token savings in turn 5 saves 12,500 tokens across the next 25 turns.

## Upgrade Instructions

```bash
git pull
cd mcp-server && npm run build
pm2 restart skillfoundry-mcp
```

The `token_usage` table is auto-created on first startup. No manual migration needed.

## Compatibility

- No breaking changes to existing tool schemas
- `concise` parameter is optional (defaults to `false`)
- Response optimizer is transparent — same data, fewer tokens
- Existing metrics and invocation tracking unchanged
